## 一、架构设计（优秀）

**亮点**

- **鉴权内核下沉到 `@tsz/shared/auth`**，web 与 admin 共用同一套 `createAuthRuntime`（http client + tokenManager + zustand store 的装配），各应用只注入 `baseUrl`/`loginPath`。`apps/web/src/lib/request.ts`、`stores/user.ts` 退化为薄壳，既是 feature 统一入口又是测试 mock 点，设计意图清晰。
- **运行环境无关的请求层**（`packages/api-client/src/http.ts`）：通过 `getToken`/`onRefresh`/`onSessionExpired` 回调注入，不耦合任何框架，纯函数 + 依赖注入，可测试性极佳。
- **端点定义即文档**（`endpoints.ts`）：按业务域组织、注释对齐后端 API 文档，类型与后端契约一一对应。
- **客户端守卫而非服务端 middleware**：经确认是有意为之（refresh cookie 的 path 限制使 middleware 拿不到凭证），`RouteGuard`/`AdminRouteGuard` 用 `hydrated` 区分"恢复中 / 未登录"，避免登录态闪烁——这是很多项目会踩的坑，这里处理得很干净。

**问题**

- **P2 — `apps/web/src/proxy.ts` 是死代码**。它形如 Next middleware（`export function proxy` + `matcher`），但 Next 只识别 `middleware.ts` 的 `middleware` 导出，文件名/导出名都不匹配，**实际从未生效**。更糟的是它读 `tsz_token` cookie 做鉴权判断，而本项目 token 是**内存态 + refresh cookie**，这个 cookie 根本不存在——即便接上也是错的。建议直接删除，避免误导后来者以为存在一层服务端门禁。
- **P3 — 主区 `(main)/layout.tsx` 不强制登录**（注释说明词表对游客开放），但 `MainNav` 直接渲染、`HomeActions` 靠 `hydrated` 占位。逻辑正确，但"哪些区域要登录"目前散落在各子 `layout` 的 `RouteGuard` 里，建议在一处文档/常量集中声明受保护路径清单，降低后续新增页面漏加守卫的风险。

## 二、代码质量（良好偏优）

**亮点**

- 注释解释**"为什么"**而非"是什么"，且不少是真正的踩坑记录。例如 `DeleteAccountForm` 里用 `window.location.replace` 而非 `router.replace` 的两条理由（重置内存态 + 规避 RouteGuard 竞态），`tokenManager` 的 visibilitychange 补偿后台标签节流，都是有价值的上下文。
- **派生态优先于 effect**：`DeleteAccountForm` 的 `channel` 用渲染期派生（`channels.includes(selectedChannel) ? ... : channels[0]`）而不是 `useEffect` 同步 state，避免多一次渲染——是 React 的正确姿势。
- 校验逻辑集中在 `packages/shared/src/validate.ts`，纯函数、正则清晰、有注释说明业务规则来源（"原型标注"）。
- 类型契约严谨（`packages/types`），`AuthResponse`/`MeResponse` 等与后端字段名（snake_case）对齐。

**问题**

- **P1 — 品牌名不一致**：`apps/web/src/app/layout.tsx`、`page.tsx`、`MainNav` 用「**天生字**」，而 `AuthBranding.tsx` 登录页大屏品牌区写的是「**天生会背**」。这是用户可见的明显出入，需统一。
- **P2 — 大量 `// TODO: 接后端` + mock 数据**（全仓约 13 处）。`useWordLists` 等 hook 用 `MOCK_WORDLISTS` 并 `setTimeout` 模拟延迟，`api.wordList.list()` 已写好却被注释。这是阶段性正常状态，但建议用统一的 feature flag / 环境开关切换 mock 与真实接口，而不是逐行注释，避免上线时漏改。
- **P3 — `http.ts` 的 `parseError` 仅识别 `{ error }` 字段**。若后端错误体结构变化（如 `{ message }` 或校验错误数组），会静默退化到 `statusText`，前端拿不到具体错误。建议与后端确认错误体规范并覆盖。
- **P3 — `HttpError` 未透传后端错误码/字段级错误**，前端只能靠 message 文本做 `translateAuthError` 映射（见各 Form 里的 `*_ERRORS` 字典）。文本匹配在后端文案变动时很脆，长期建议改为错误 code 映射。

## 三、代码性能（良好）

**亮点**

- **并发刷新去重**：`tokenManager.refreshTokens` 用单例 `refreshingPromise`，多个 401 同时触发只发一次 `/auth/refresh`，避免刷新风暴与竞态。
- **主动刷新定时器自滚动**（提前 30s）+ **visibilitychange 兜底**，对"后台标签 setTimeout 被节流"这一真实问题做了补偿。
- token 清除时同步 `clearTimeout`，避免登出后定时器还在跑。
- zustand 选择器按字段订阅（`useUserStore((s) => s.user)`），`MainNav` 注释也说明了"`hasRole` 是稳定引用不触发重渲染"，对重渲染有意识。

**问题**

- **P3 — `redirectToLogin`/`refreshTokens` 失败统一跳 `loginPath`**，但 401-retry 与定时器两条路径都可能并发触发跳转，极端情况下可能重复 `window.location.href`。影响轻微（浏览器会去重导航），可不处理，但值得知道。
- **P3 — 列表渲染无虚拟化**。当前 mock 数据量小无影响；接入真实词表/词库分页后，若单页量大需考虑 `react-window` 之类。现在不用做。

## 四、网站性能（中等，最大优化空间）

- **客户端组件偏多**：`apps/web` 25 个 tsx 中 17 个标了 `"use client"`。鉴权强相关的确需要客户端，但部分纯展示页（如词表卡片列表）本可用 Server Component + 流式渲染减少首屏 JS。当前 `WordListBrowser` 整个是 client + TanStack Query，首屏需等 JS 加载 + 查询。
- **未使用 Next 的资源优化**：全仓 `next/image` 使用 **0 处**，品牌图标用 emoji（📖）。若后续有真实插图/头像，应走 `next/image`。也未见 `next/font` 字体优化配置。
- **首屏鉴权瀑布**：`Providers → SessionRestorer → refreshTokens() → me()` 是串行两跳网络请求才能确定登录态，期间各守卫显示"加载中…"。可接受，但首屏 TTI 受这两跳影响；可考虑后端把 `/auth/refresh` 响应顺带返回 user 信息，省掉 `/me` 一跳。
- **`ReactQueryDevtools` 在 `providers.tsx` 无条件引入**。需确认生产构建会 tree-shake 掉（TanStack 的 devtools 生产环境通常是 no-op，但建议显式 `process.env.NODE_ENV` 守卫或动态 import 确保不进包）。
- **缺少可量化基线**：未见 `@next/bundle-analyzer` 或 Lighthouse/Web Vitals 上报。建议在 CI 增加一次 `next build` 的产物体积快照，作为性能回归门槛（你已有 turbo 缓存基础，加这个成本很低）。
- 正向项：`next.config.mjs` 用 `transpilePackages` 直接编译 workspace 包、`rewrites` 代理后端，配置干净；CI 已缓存 turbo 与 Playwright 浏览器。

## 五、UI 设计（鉴权流精致，核心功能待建）

- **鉴权流（登录/注册/找回密码/注销）视觉成熟**：统一的 `AUTH_INPUT_CLASS`、`rounded-full` 主按钮、`blue-600` 主色 + 中性灰阶、禁用态 `opacity-40`、`transition-colors`、倒计时"60s 后重发"等微交互齐全，注销页还有居中标题占位、危险提示红字，细节到位。
- **核心学习功能页是占位骨架**：`PracticeBoard`/`CoinsPanel`/`TaskManager`/`ClassManager`/`StatsDashboard` 基本是一句话 + 空 `Card`（"…在此实现"）。这是 UI 完成度的主要缺口，符合"先搭骨架"的节奏，但离"美观"还很远。
- **P1 — 品牌名「天生字 / 天生会背」不一致**（见代码质量节），属用户可见问题。
- **P3 — 缺设计系统沉淀**：`globals.css` 只定义了 `--background`/`--foreground` 两个变量且未实际用于主题（被 layout 的 `bg-gray-50` 覆盖），颜色/圆角/间距散落在各组件的 Tailwind class。`packages/ui` 仅 `Button`/`Card` 两个原子组件。建议把 `blue-600`、`rounded-full`、输入框样式等收敛进 `@tsz/ui` 或 Tailwind theme，保证 web/admin 视觉一致、便于后续做暗色/换肤。
- **P3 — 无暗色模式 / 无明显无障碍体系**：有 `aria-hidden` 占位等零星处理，但未见整体 a11y 约定（focus ring、对比度、`aria-live` 错误提示）。

## 六、测试与工程化（优秀）

- **覆盖率门槛分层**：`packages/**` 要求 100%（语句/分支/函数/行），`apps/*/src` 90%，且 `vitest.config.ts` 精确圈定纳入范围（业务逻辑层）与排除项（路由壳/纯静态），思路清晰。
- **本地与 CI 对齐**：lefthook `pre-push` 跑 `test:cov` + typecheck + e2e，注释明确说明"避免本地全绿、推 CI 才红"。
- **CI 设计精细**：最小权限（`contents: read`）、PR 取消旧任务但 main 不取消、commitlint 兜底 `--no-verify`、Playwright 浏览器按版本缓存、e2e 独立 job 不阻塞单测反馈、失败上传报告。
- 43 个测试文件覆盖鉴权全链路、表单交互、guard 跳转、http 重试。

**小建议**

- **P3** — e2e 用 `mockApi.ts` 拦截后端，属契约 mock；真实后端接入后建议补一条**对真实接口**的冒烟（至少登录），否则前后端契约漂移测不出来。

## 七、安全（基本到位）

- access token 仅存内存、不落 localStorage，refresh 走 httpOnly cookie（`credentials: "include"`），是当前推荐的前端 token 存储方案，能规避 XSS 直接窃取 token。
- 公开端点 `skipAuth` 避免旧 token 污染登录请求；注销账号用 `channel + code` 二次验证且验证码发往**在档**联系方式（防止冒用），后端语义设计严谨。
- `env.ts` 用 zod 校验 `NEXT_PUBLIC_*`，对环境变量做了运行时保证。
- **P3** — 死代码 `proxy.ts` 读不存在的 `tsz_token` 做鉴权，若有人误信会形成"看起来有门禁实则没有"的安全错觉，删除即可。

## 结论

**底子非常好**，鉴权内核、分层、测试与 CI 是这套代码的核心资产，可直接作为团队标杆。当前最值得马上处理的是 **P1 品牌名不一致** 和 **P2 删除死代码 `proxy.ts`**——成本极低、收益明确。中期重点是**接入真实后端 + 实现核心学习功能页**，并借机补上**网站性能基线（SSR 下推 + 产物体积门槛）**与**设计系统沉淀**这两块目前最薄的环节。
