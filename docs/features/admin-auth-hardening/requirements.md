# 后台登录加固对接 需求评估

> 对接文档：`../docs/admin-auth-hardening-frontend-integration.md`（后端 PR #48「后台登录加固」四段合一，
> 接口均 ✅ 已上线生产可联调）。契约以 `../tsz-go/docs/openapi.yaml` 标签 `Admin (auth)` / `Admin (accounts)` 为准。

## 背景与目标

后端对**管理控制台**（admin，与 web 学员/教师完全独立的账号体系）做了一轮登录加固，涉及 4 项能力：
密码强度策略升级、强制改密闭环、账号锁定、超管重置密码。其中部分能力前端已在既往迭代
（提交 `7a39a02 feat(admin): 后台用户管理 + 管理员管理（含强制改密）`）落地，本次是**对齐对接文档、补齐剩余缺口**。

**成功长什么样**：被超管重置密码的管理员，无论从登录还是刷新页面进入，都能被稳定引导到改密页完成改密、
不再出现「每个接口都 403 但前端无所适从」的死锁；连续登录失败被锁定的账号能看到明确的 423 锁定提示而非英文原文；
管理员可自助修改密码。

## 目标端

**admin** 单端（管理后台，Vite + React Router + antd v6）。web 端不涉及本次任何改动
（后台与 web 是两套独立账号体系，鉴权代码不得混用）。共用底层（`@tsz/api-client` 的 `http.ts`、
`@tsz/shared/auth`）的改动必须对 web 保持向后兼容、零影响。

## 用户故事 / 使用场景

1. **被重置管理员·登录即改密**（已实现，保留）：超管重置了管理员 A 的密码 → A 用临时密码登录 →
   登录响应 `must_change_password=true` → 强制进改密流程 → 用临时密码作 current_password 提交新密码 →
   204 成功 → 进入控制台。
2. **被重置管理员·刷新后重新发现**（新增，关键）：A 在改密前刷新了页面 / 重开标签页 → refresh 不带
   `must_change_password` 标记 → A 进入控制台壳后任一后台接口返回 `403 code:must_change_password` →
   前端全局拦截，强制跳改密页 → 改密成功后正常使用。
3. **账号被锁定**（新增）：某账号连续登录失败 5 次 → 后端锁定 15 分钟 → 此后登录（即使密码正确）返回
   **423** → 前端显示「已锁定，请 15 分钟后再试」并**置灰登录按钮一段时间**，区别于 401 凭证错误。
4. **日常自助改密**（新增）：已登录管理员在顶栏进入「修改密码」→ 填当前密码 + 新密码 → 204 成功提示。

## 功能范围

**做（本次范围）**：

- **A. 登录 423 账号锁定分支**：文案「账号已锁定，请 15 分钟后再试」+ 登录按钮临时置灰（一段时间，非本地数失败次数）。
- **B. 全局 403 强制改密拦截**：在共用请求层加一条「`status===403 && code==='must_change_password'`」分支，
  经 admin 专属回调触发跳改密页，对**所有**后台请求生效（不只登录时）。含 `HttpError` 扩展读取 `code` 字段。
- **C. 独立改密页 + 自助入口**：新增 `/change-password` 路由页，同时服务「强制改密（403/登录触发）」与
  「日常自助改密（顶栏入口）」两种场景；顶栏加「修改密码」入口。
- **D. 收尾对齐**：确认创建/改密表单密码本地校验与后端策略（≥12、非纯数字等）一致，400 一律展示后端文案。

**不做（划走边界）**：

- 不碰 web 端任何鉴权代码。
- 不在前端本地数登录失败次数决定何时提示锁定——以后端 423 为唯一事实来源。
- 不实现「常见弱口令 / 含手机号」的本地预校验穷举（后端为准，本地只做基础提示；对接文档明确本地校验为可选优化）。
- 不改后端：吊销会话、审计、密码哈希等均后端已实现，前端只对接。
- 不动已实现且符合对接文档的 §1（登录 must_change 弹窗）、§5（重置密码弹窗）现有交互，除非被 C 的改密页收编。

## 约束与边界

- **权限/角色**：改密接口面向当前登录管理员（admin / super_admin 皆可）；重置密码超管专属（已实现，不动）。
- **契约稳定性**：403 分支**必须用 `code` 判断，不得用 `error` 文案**（文案可变，code 是稳定契约）。
- **改密可达端点**：`must_change_password` 未清前，仅 change-password / logout / logout-all 三个接口不被 403 守卫拦截；
  改密页与登出逻辑不能触发全局 403 跳转的自循环。
- **423 无 `code` 字段**：靠 HTTP 状态码判断。
- **共用层兼容**：`http.ts` / `createHttpClient` 为 web+admin 共用，新增回调必须可选，web 不传即维持原行为。
- **密码策略**：新密码 ≥12 位、非纯数字、非弱口令、不含手机号、（改密）与当前不同；违反后端返回 400，前端展示后端 `error`。

## 验收标准

- [ ] 登录连续失败被锁：登录返回 423 时显示中文锁定提示，登录按钮临时置灰，与 401/429/403 文案可区分。
- [ ] 被重置管理员**刷新页面**后，进入控制台触发任一后台请求即被拦截跳转到 `/change-password`，不出现死锁。
- [ ] 全局 403 拦截仅对 `code==='must_change_password'` 生效，其它 403（如非超管越权）不误跳。
- [ ] `/change-password` 页在「强制改密」与「日常自助改密」两种场景下都能提交，处理 204 / 401 / 400 三态。
- [ ] 顶栏存在「修改密码」入口，进入自助改密页；改密 204 成功后有明确成功反馈。
- [ ] 改密 / 登出 / 全端登出接口本身不会被全局 403 拦截误跳（无自循环）。
- [ ] 创建管理员、改密表单密码本地校验 ≥12 位等与后端一致，400 时展示后端文案。
- [ ] web 端鉴权行为不受共用层改动影响（回归：登录/刷新/登出正常）。
- [ ] `pnpm typecheck` / `pnpm lint` / `pnpm test:cov` 全绿，达 admin 90% / packages 100% 覆盖率门槛。

## 已定决策（评审通过）

1. **改密页形态 → 独立路由页 `/change-password`**（架构强制项）：全局 403 拦截在无状态请求层触发，
   唯一干净落点是 `window.location` 整页跳转到可路由地址；弹窗无法作为跳转目标，且会把路由/store 引用
   泄进请求层。独立页同时服务强制改密与自助改密。
2. **强制改密期会话恢复 → 接受短暂闪现**：刷新后靠 403 重新发现，其间 console 壳短暂渲染再整页跳转，
   干净利落、可接受；不在守卫层前置探测（列后续优化）。
3. **423 置灰 → 置灰 + 输入变更即恢复，不做倒计时**：后端 423 不返回锁定剩余秒数且计数跨会话，
   固定倒计时是假精度会误导；显示锁定文案 + 置灰登录按钮，用户改动账号/密码输入时恢复可点，
   仍锁定则再拿 423（事实来源）。另附**后端建议**（423 带 `Retry-After`/剩余秒数）供转达，本次不依赖。
4. **范围边界**：不做弱口令/含手机号本地穷举预校验（后端 400 为准）；不碰 web 端。
